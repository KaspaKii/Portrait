# Portrait Compiler — First Slice Evidence

**Stichting Kii Foundation — kaspa-compliance-patterns**

> **Status:** Pre-production. The Portrait compiler (`portrait/portrait/`) is an internal
> Kii Foundation tool, open-sourced, that compiles a high-level Portrait DSL into silverscript
> covenants. This document records the first working compiler slice (M0 target: `counter.portrait`).

---

## 1. Pipeline demonstrated

```
counter.portrait
  → portrait-syntax::parse    (AST)
  → portrait-ir::lower        (Cartoon IR)
  → portrait-project::project (CovenantModel per role)
  → portrait-emit::emit       (SilFile: Counter.sil + Counter_ctor.json)
  → silverc --ctor Counter_ctor.json Counter.sil
  → Counter.json              (compiled covenant artifact, exit 0)
```

All five stages implemented and verified in a single `portrait build counter.portrait` invocation.

---

## 2. Source: counter.portrait

```portrait
pragma portrait ^0.1.0;

app Counter {
  role counter {
    param int start;
    state { int value; }

    #[covenant(mode = transition)]
    entrypoint function bump(int delta) : (int value) {
      return value + delta;
    }
  }

  lifecycle { live -> live via counter.bump; }
  invariant no_undeclared_state;
}
```

---

## 3. Emitted: Counter.sil

```silverscript
pragma silverscript ^0.1.0;

// Generated by portrait build
contract Counter(int max_ins, int max_outs, int start) {
    int value = start;

    #[covenant(binding = cov, from = max_ins, to = max_outs, mode = transition)]
    function bump(State[] prev_states) : (State[]) {
        return(prev_states);
    }
}
```

## 7. Known limitations (M0)

1. **State field initialization is positional.** The i-th state field is initialized from the i-th user constructor param. This works for Counter (1 field, 1 param) but breaks for contracts where params and fields don't align positionally.

2. **Transition body is a stub.** The emitted `bump` function returns `prev_states` unchanged — it carries forward the state identity but doesn't implement the `value + delta` arithmetic. Full expression lowering is M1 work.

3. **No entry-point argument threading.** The Portrait `bump(int delta)` takes a `delta` arg, but the M0 emit doesn't thread it through to the silverscript function signature. Full arg lowering is M1 work.

4. **Multi-role apps not tested.** The `project()` function handles multi-role apps by suffixing the role name, but this path is untested at M0.

5. **No deploy.** The compiled `Counter.json` artifact is a covenant script. Deploying it to TN10 requires funding a UTXO with the genesis state (a Kaspa `send` transaction). This is out of scope for the M0 slice; TN10 deploy evidence for the KTT-profile patterns already exists (`[KCP-KTT-003]`).

---

## 8. What this proves

- The Portrait DSL → silverscript pipeline is end-to-end functional for the M0 target.
- The Cartoon IR correctly models the Counter lifecycle (`live → live via counter.bump`).
- The emitted silverscript is accepted by the `silverc` compiler (Kaspa's official covenant compiler).
- The pipeline is reproducible: `portrait build counter.portrait` produces a deterministic `Counter.sil`.

The next slice (M1) would thread arguments through the emit, implement expression lowering (`value + delta`), and add multi-role projection tests.
